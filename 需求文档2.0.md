你是资深全栈工程师，请根据以下需求实现一个 MVP 版本的 AI Story Foundry。

项目目标：
构建一个 AI Story Engine。用户输入故事创意后，系统自动生成 Story Bible，并基于 Story Bible 生成动态规则、动态 UI、多人房间、AI GM、AI NPC、World State 和动态结局。

核心原则：
1. Story Bible 是唯一故事数据源。
2. World State 是唯一真实世界状态源。
3. Rule Engine 是唯一状态修改入口。
4. AI GM 可以读取完整 Story Bible，但不能直接修改 World State。
5. AI NPC 不能读取完整 Story Bible，只能读取 Knowledge Filter 生成的局部视野。
6. 玩家交互支持 Chat、Suggested Actions、Free Action 三种方式，但底层统一转换成 Structured Action。
7. 前端 UI 不由 AI 生成代码，而是通过 Widget Registry 根据 ui_config 动态拼装。

技术栈建议：
- Frontend: Next.js + React + TypeScript + Tailwind CSS
- Backend: Node.js + NestJS 或 Next.js API Routes
- Realtime: WebSocket 或 Socket.IO
- Database: PostgreSQL，MVP 可先用内存或 JSON 文件模拟
- AI 调用层先抽象成接口，可用 mock provider

请优先实现 MVP，不要一次性做复杂商业化功能。

====================
二、模块实现要求
====================

请实现以下模块：

1. Story Input Module
接收用户故事创意，转换为标准 StorySeed。

2. Story Analyzer
分析故事是否适合多人互动演绎。
输出：
- playable
- playable_score
- features
- missing_elements
- suggested_fixes

features 示例：
{
  has_conflict: true,
  has_mystery: true,
  has_factions: true,
  has_relationships: false,
  has_survival_pressure: false,
  has_resource_competition: true
}

3. Story Adapter
如果故事可玩性不足，自动补全：
- 角色目标
- 秘密目标
- 阵营
- 冲突
- 事件
- 章节
- 结局条件

4. Story Bible Generator
根据 StorySeed 和 Adapter 结果生成 Story Bible。

MVP 可以先 mock 一个西幻权谋悬疑故事：
标题：失落圣杯之夜
角色：王子、圣女、刺客、骑士
NPC：大法师、老国王、主教
核心事件：圣杯失窃、圣殿调查、古神仪式
结局：王国得救、古神苏醒、王权崩塌、刺客革命成功

5. Story Bible Validator
校验：
- 每个角色必须有 id/name/goal
- 每个 NPC 必须有 goal/knowledge_scope
- 每个 Event 必须有 trigger/effects
- 每个 Ending 必须有 conditions
- 每个 Metric 必须有 id/type/initial
- ui_config.widgets 必须存在于 WidgetRegistry
- NPC 不得拥有 full_story_access
- 玩家角色数量至少 2 个
- 结局至少 2 个

6. Feature Extraction
从 Story Bible 中提取互动机制，不按题材分类，而按机制分类。

7. Rule Pack Selection
根据 features 选择：
- FactionPack
- MysteryPack
- RelationshipPack
- SurvivalPack
- ResourcePack

MVP 至少实现：
- FactionPack
- MysteryPack
- RelationshipPack

8. Dynamic Metric Generation
根据 Story Bible 生成 metrics。
不要写死好感度、怀疑值、势力值。
但 MVP 可以内置：
- kingdom_stability 王国稳定度
- truth_progress 真相进度
- holy_grail_influence 圣杯影响力
- faction_power 阵营影响力
- trust 信任度
- suspicion 怀疑值

9. UI Config Generation
生成：
{
  theme,
  layout,
  widgets,
  display_metrics
}

10. Widget Registry
前端提前实现组件：
- ChatPanel
- NarrativePanel
- RolePanel
- MetricPanel
- EventPanel
- FactionPanel
- EvidencePanel
- SuggestedActionsPanel
- FreeActionInput
- PlayerInfoPanel

AI 只输出 widget key，前端只渲染 registry 中存在的组件。

11. UI Builder
根据 story_bible.ui_config.widgets 动态渲染页面。

12. Room System
实现：
- 创建房间
- 加入房间
- 选择角色
- 开始故事
- 房间状态管理

13. World State Engine
维护真实状态。
任何状态变化必须通过 Rule Engine。

14. AI GM Module
AI GM 可读取完整 Story Bible 和 World State。
负责：
- 生成叙事
- 解释行动结果
- 推进节奏
- 生成章节过渡
- 生成结局叙事

限制：
- 不得直接修改 World State
- 只能输出 narration、suggested_events、revealed_information

15. NPC Knowledge Filter / Perspective Builder
为每个 NPC 构造局部视野。

NPC 可见：
- 自己目标
- 自己秘密目标
- 自己记忆
- 自己已知事实
- 公开事件
- 自己关系状态

NPC 不可见：
- 完整 Story Bible
- 最终真相
- 其他 NPC 秘密
- 隐藏事件触发条件
- 未来章节
- 结局条件
- 玩家私密行动
- 未公开线索

16. AI NPC Module
NPC 基于局部视野输出 Action Proposal。

输出示例：
{
  npc_id,
  intention,
  action_type,
  target,
  reasoning_visible,
  requires_rule_check: true
}

17. Player Interaction Layer
这是重点模块。

玩家交互必须支持三种方式：

A. Roleplay Chat
用于角色扮演和交流。
默认不改变 World State。
示例：
“圣女，你昨晚为什么出现在圣殿？”

B. Suggested Actions
系统根据当前位置、剧情、角色身份推荐行动按钮。
示例：
[调查圣殿]
[询问守卫]
[查看案发现场]

C. Free Action
玩家自由输入真实行动。
示例：
“我要收买守卫，让他告诉我昨晚发生了什么。”

无论玩家点击按钮还是自由输入，最终都必须转换为 StructuredAction。

18. Action Parser
将玩家输入转换成 StructuredAction。

输入：
“我要假扮修士潜入教堂地下室，偷听主教和大法师的谈话。”

输出：
{
  action_type: "investigate",
  target: "cathedral_basement",
  method: "stealth_disguise",
  intent: "eavesdrop",
  risk_level: "high"
}

20. Rule Engine
负责：
- 权限检查
- 成功率判断
- 指标更新
- flag 更新
- 关系更新
- 事件状态更新
- 线索获取
- NPC 行动审核

输入 StructuredAction。
输出 RuleResult。

RuleResult 示例：
{
  success: true,
  state_updates: [
    {
      type: "add_known_fact",
      target: "player_001",
      fact_id: "ancient_map_fragment"
    },
    {
      type: "metric_change",
      metric: "truth_progress",
      delta: 10
    }
  ],
  public_result: "你发现了一张残缺地图。",
  private_result: "地图上标记着古代圣杯遗迹。"
}

21. Event Trigger System
根据 World State 自动触发事件。

22. Story Controller
根据：
- turn
- chapter
- events
- flags
- metrics

推进章节。

23. Ending Judge
根据 World State 判断结局。
AI GM 只负责结局叙事，不负责结局判定。

====================
四、玩家交互详细流程
====================

游戏运行时流程：

1. AI GM 输出当前剧情叙事。
2. 玩家阅读 NarrativePanel。
3. 玩家可以在 ChatPanel 自由 RP 发言。
4. 玩家可以点击 Suggested Actions。
5. 玩家也可以在 FreeActionInput 输入自定义行动。
6. 系统将行动送入 Action Parser。
7. Action Parser 输出 StructuredAction。
8. Rule Engine 进行权限、风险、成功率和后果结算。
9. World State Engine 应用状态更新。
10. Event Trigger System 检查是否触发新事件。
11. NPC Knowledge Filter 为每个 NPC 生成局部视野。
12. AI NPC Planner 生成 NPC Action Proposal。
13. NPC Action Proposal 进入 Rule Engine。
14. World State 再次更新。
15. Story Controller 判断是否推进章节。
16. Ending Judge 判断是否达成结局。
17. AI GM 生成新的叙事。
18. 进入下一轮。

注意：
Chat Message 默认不触发规则。
Action 才进入 Rule Engine。
如果玩家在 Chat 中说了明显行动，例如“我现在拔剑刺杀国王”，系统应提示：
“这看起来是一个行动，是否作为正式行动提交？”
