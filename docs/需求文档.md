# AI Story Foundry 需求文档

## 1. 项目背景

传统剧本杀、跑团和互动小说的内容生产成本高，规则和叙事通常强绑定单一剧本。AI Story Foundry 的目标是构建一个可复用的 AI 互动叙事引擎，让用户输入任意故事创意或剧本文本后，系统自动生成可运行的多人故事世界，并支持玩家以角色扮演、推荐行动和自由行动推动剧情。

## 2. 项目目标

构建 MVP 版本的 AI Story Engine，实现：

- 用户输入故事创意或完整剧本。
- 系统生成 Story Bible。
- 系统基于 Story Bible 生成动态规则、动态 UI、多人房间、AI GM、AI NPC、World State 和多结局。
- 玩家可以创建房间、选择角色、开始故事、聊天、执行行动并进入结局。
- NPC 可以基于有限认知主动行动。
- 所有状态变化必须经过 Rule Engine，保证 AI 不直接篡改世界状态。

## 3. 用户角色

| 用户角色 | 说明 |
| --- | --- |
| 房主 | 创建故事和房间，邀请其他玩家，开始故事 |
| 玩家 | 加入房间，选择角色，进行聊天和行动 |
| AI GM | 生成叙事、解释结果、给出推荐行动，不直接修改状态 |
| AI NPC | 基于有限视野生成行动提案，不读取完整故事真相 |

## 4. 核心业务流程

### 4.1 创建故事流程

```text
用户进入首页
  -> 点击创建故事
  -> 输入故事创意/完整剧本/结构化字段
  -> 系统分析可玩性
  -> 系统生成 Story Bible
  -> 系统校验 Story Bible
  -> 用户查看摘要
  -> 创建房间
```

### 4.2 加入和开局流程

```text
房主分享房间号
  -> 玩家输入房间号与昵称
  -> 进入房间大厅
  -> 玩家选择角色
  -> 房主开始故事
  -> 系统初始化 World State
  -> 进入游戏主界面
```

### 4.3 游戏回合流程

```text
AI GM 输出当前叙事
  -> 玩家阅读故事
  -> 玩家聊天或执行正式行动
  -> 行动解析为 StructuredAction
  -> Rule Engine 结算
  -> World State 更新
  -> NPC 根据局部视野行动
  -> 事件触发与章节推进
  -> Ending Judge 判断结局
  -> AI GM 生成下一段叙事
```

## 5. 功能需求

### 5.1 首页

优先级：P0

需求：

- 用户可以进入故事生成页。
- 用户可以输入房间号加入已有房间。
- 用户可以快速加载 Demo 故事。
- 系统应保存玩家本地身份，避免刷新后丢失。

验收标准：

- 点击创建故事后进入 `/generate`。
- 输入有效房间号和昵称后进入对应房间。
- 点击 Demo 后进入 Demo 故事生成页。

### 5.2 故事输入与生成

优先级：P0

需求：

- 支持单段故事创意输入。
- 支持完整剧本文本导入。
- 支持题材、开场、结局方向、角色、世界观等结构化字段。
- 系统将输入归一化为 StorySeed。
- 系统生成 Story Bible。
- 系统展示 Story Bible 摘要、角色、NPC、阵营、结局、玩法模块和校验结果。

验收标准：

- 输入故事后可以生成 Story Bible。
- 校验通过的 Story Bible 可以创建房间。
- 校验不通过时显示错误或可玩性建议。

### 5.3 Story Bible

优先级：P0

需求：

Story Bible 必须包含：

- `id`
- `title`
- `version`
- `world_setting`
- `roles`
- `npcs`
- `factions`
- `chapters`
- `events`
- `endings`
- `metrics`
- `rules`
- `knowledge`
- `ui_config`

扩展需求：

- 支持运行时模块 `runtime_modules`。
- 支持角色模型、阵营模型、关系图谱、知识图谱和胜利条件。

验收标准：

- Story Bible 能够被校验器检查。
- Story Bible 能够驱动房间开局、UI 渲染和规则结算。

### 5.4 Story Bible 校验

优先级：P0

需求：

- 校验角色是否有 id、name、goal。
- 校验 NPC 是否有 goal 和有限知识范围。
- 校验 Event 是否有 trigger/effects。
- 校验 Ending 是否有 conditions。
- 校验 Metric 是否有 id、type、initial。
- 校验 `ui_config.widgets` 是否存在于 Widget Registry。
- 禁止 NPC 拥有完整故事访问权。
- 至少包含 2 个玩家角色。
- 至少包含 2 个结局。

验收标准：

- 校验失败时不允许创建正式房间。
- 校验错误包含字段和原因。

### 5.5 房间系统

优先级：P0

需求：

- 创建房间。
- 加入房间。
- 选择角色。
- 防止多个玩家选择同一角色。
- 房主开始故事。
- 管理房间状态。

验收标准：

- 创建房间后生成房间号。
- 玩家可通过房间号加入。
- 所有已加入玩家选择角色后可开局。
- 开局后生成 World State。

### 5.6 World State

优先级：P0

需求：

World State 必须记录：

- 故事 id 与房间 id。
- 当前章节。
- 当前 turn。
- flags。
- metrics。
- events。
- relationships。
- locations。
- knowledge_state。
- character_states。
- faction_states。
- communication_state。
- balance_state。
- turn_state。

验收标准：

- 所有状态变化通过 Rule Engine 生成的 StateUpdate 应用。
- 玩家视图从 World State 派生。
- 事件、章节、结局均基于 World State 判断。

### 5.7 玩家交互

优先级：P0

需求：

- 支持 Roleplay Chat。
- 支持 Suggested Actions。
- 支持 Free Action。
- 聊天默认不改变 World State。
- 明显行动型聊天应提示玩家转为正式行动。
- 所有正式行动必须转为 StructuredAction。

验收标准：

- 推荐行动点击后进入规则结算。
- 自由行动输入后进入解析和规则结算。
- 聊天消息可以发送和显示。
- 行动执行时有 pending 反馈。

### 5.8 Action Parser

优先级：P0

需求：

- 将自然语言行动解析为 `StructuredAction`。
- 识别行动类型、目标、方式、意图、风险等级。
- 支持 AI 解析和本地兜底解析。
- 支持题材禁用行动，例如非死亡题材不直接执行刺杀。

验收标准：

- “我潜入地下室偷听主教谈话”应被解析为调查/窃听类行动。
- “我说服守卫开门”应被解析为社交/说服类行动。
- 无法识别时给出安全默认行动。

### 5.9 Rule Engine

优先级：P0

需求：

- 检查行动权限。
- 计算成功率和风险。
- 生成 StateUpdate。
- 更新指标、flag、关系、知识、证据、位置、事件状态。
- 处理 NPC 行动审核。
- 处理角色死亡、失踪、囚禁、失败、受挫等状态。
- 处理题材安全后果。

验收标准：

- Rule Engine 返回 `RuleResult`。
- World State 只应用 RuleResult 中的状态更新。
- AI GM 和 AI NPC 不能直接修改 World State。

### 5.10 AI GM

优先级：P1

需求：

- 读取完整 Story Bible 和 World State。
- 生成开场叙事。
- 解释玩家行动结果。
- 总结 NPC 行动影响。
- 生成章节过渡叙事。
- 生成结局叙事。
- 生成推荐行动。

限制：

- 不直接修改 World State。
- 不输出隐藏指标 key、内部 id 或未解锁剧透。

验收标准：

- 玩家行动后能得到新的 GM 叙事。
- GM 叙事能反映指标和事件变化。
- 推荐行动能围绕当前 World State 推进。

### 5.11 AI NPC

优先级：P1

需求：

- 每个 NPC 拥有独立 Runtime State。
- NPC 只读取局部视野。
- NPC 不能读取完整 Story Bible、最终真相、其他 NPC 秘密、未来章节和结局条件。
- NPC 可主动生成行动提案。
- NPC 行动必须交给 Rule Engine 审核。
- NPC 可阻碍、误导、隐藏证据、影响指标、推进计划。

验收标准：

- NPC 能在玩家回合后按频率行动。
- NPC 行动能影响故事张力。
- NPC 不会无限锁定同一玩家。
- NPC 不会永久删除关键线索。

### 5.12 事件系统

优先级：P1

需求：

- 根据 World State 自动触发事件。
- 支持 flag、metric、已知事实等触发条件。
- 事件效果支持设置 flag、修改指标、添加知识、揭示事件、改变位置。

验收标准：

- 满足条件后事件自动触发。
- 事件触发结果写入 World State。
- 玩家只看到已触发或已解锁事件。

### 5.13 章节推进

优先级：P1

需求：

- 根据 turn、chapter、events、flags、metrics 推进章节。
- 章节推进后记录章节启动标记。

验收标准：

- 满足章节条件后自动切换章节。
- GM 叙事能反映章节变化。

### 5.14 结局判断

优先级：P1

需求：

- 根据 World State 判断结局。
- 支持多个结局和优先级。
- AI GM 只负责结局叙事，不负责结局判定。
- 支持阵营胜利和个人胜利。

验收标准：

- 满足结局条件后返回结局。
- 前端跳转结局页。
- 结局页显示结局标题与叙事。

### 5.15 动态 UI

优先级：P0

需求：

- 前端通过 Widget Registry 渲染 UI。
- AI 只能输出 widget key，不能输出前端代码。
- 不存在的 widget key 应安全降级。

必须支持 Widget：

- `ChatPanel`
- `NarrativePanel`
- `RolePanel`
- `MetricPanel`
- `EventPanel`
- `FactionPanel`
- `EvidencePanel`
- `SuggestedActionsPanel`
- `FreeActionInput`
- `PlayerInfoPanel`

验收标准：

- Story Bible 的 `ui_config.widgets` 能决定页面面板。
- Widget key 不存在时不导致页面崩溃。

## 6. 非功能需求

### 6.1 安全与边界

- NPC 不能读取完整 Story Bible。
- AI GM 和 AI NPC 不能直接修改 World State。
- 所有正式行动必须结构化。
- 所有状态变化必须通过 Rule Engine。
- 玩家视图不得泄露隐藏事实、内部 id、隐藏指标和结局条件。

### 6.2 可维护性

- 使用 TypeScript 类型定义核心数据。
- 引擎模块应保持单一职责。
- 新增题材模块时应通过 runtime_modules 和规则适配扩展。
- 新增 UI 面板时必须注册 Widget。

### 6.3 可扩展性

- AI Provider 应保持可替换。
- 内存存储后续应可迁移到数据库。
- Realtime 能力后续可接入 Socket.IO。
- Story Bible 应支持版本迁移。

### 6.4 可用性

- 玩家行动提交后应立即看到处理中反馈。
- 行动失败应恢复推荐行动列表。
- AI 失败时应明确提示，不用错误故事冒充真实生成结果。
- Demo 应始终可用于快速体验。

## 8. 当前 MVP 验收范围

MVP 应至少满足：

1. 可以运行 Demo 故事“失落圣杯之夜”。
2. 可以创建房间。
3. 可以选择角色。
4. 可以开始故事。
5. 可以显示 GM 叙事。
6. 可以点击推荐行动。
7. 可以输入自由行动。
8. 可以解析行动。
9. Rule Engine 可以更新 World State。
10. NPC 可以基于有限认知行动。
11. 可以触发事件。
12. 可以推进章节。
13. 可以判断结局。
14. 可以进入结局页。

## 9. 后续需求

- 房间与世界状态持久化。
- 多玩家实时同步。
- 房主权限管理。
- 断线重连。
- 旁观者模式。
- 回放和行动日志。
- Story Bible 编辑器。
- 更严格的 schema 校验。
- 多模型 Provider 管理。
- 自动化测试覆盖核心规则。
