# AI Story Foundry 开发文档

## 1. 项目概述

AI Story Foundry 是一个基于 Next.js 的多人 AI 互动叙事引擎。用户输入故事创意或完整剧本后，系统生成 Story Bible，并围绕 Story Bible 构建动态规则、动态 UI、多人房间、AI GM、AI NPC、World State、事件触发和多结局流程。

当前项目版本为 `1.6.0`，MVP 已实现从故事生成到房间开局、玩家行动、规则结算、NPC 干预、GM 叙事、事件推进和结局判断的主闭环。

## 2. 技术栈

| 层级 | 技术 |
| --- | --- |
| 前端 | Next.js 14、React 18、TypeScript、Tailwind CSS |
| 后端 | Next.js App Router API Routes |
| 状态存储 | MVP 使用服务端内存 Map，挂载在 `globalThis` |
| AI 接入 | OpenAI-compatible Chat Completions Provider，默认兼容 DeepSeek |
| 类型校验 | TypeScript 类型约束，部分结构使用本地校验器 |
| 实时能力 | `socket.io` 已作为依赖预留，当前主链路仍使用 HTTP API |

## 5. 核心设计原则

1. Story Bible 是唯一故事数据源。
2. World State 是唯一真实世界状态源。
3. Rule Engine 是唯一状态修改入口。
4. AI GM 可以读取完整 Story Bible 与 World State，但不能直接修改 World State。
5. AI NPC 只能读取 Knowledge Filter 构造的局部视野。
6. 玩家聊天、推荐行动、自由行动最终都必须转换为 `StructuredAction`。
7. 前端 UI 不由 AI 生成代码，而是通过 Widget Registry 根据 `ui_config.widgets` 动态渲染。

## 6. 核心模块说明

| 模块 | 文件 | 说明 |
| --- | --- | --- |
| 故事生成 | `src/engine/storyBibleGenerator.ts` | 根据 StorySeed 生成 Story Bible |

| 剧本解析 | `src/engine/fullTextScriptParser.ts` | 从完整剧本文本提取结构化种子信息 |

| Story Bible 校验 | `src/engine/storyBibleValidator.ts` | 校验角色、NPC、事件、结局、指标、Widget 配置 |
| World State | `src/engine/worldStateEngine.ts` | 创建状态、应用状态更新、读取指标和知识 |
| 行动解析 | `src/engine/actionParser.ts` | 将推荐行动/自由输入解析为 StructuredAction |
| 规则结算 | `src/engine/ruleEngine.ts` | 权限、风险、成功率、指标、关系、知识、角色状态结算 |
| NPC 视野 | `src/engine/npcKnowledgeFilter.ts` | 为 NPC 构造受限局部视野 |
| NPC 计划 | `src/engine/npcPlanner.ts` | 基于局部视野生成 NPC Action Proposal |
| NPC 回合 | `src/engine/npcTurnSystem.ts` | 定时运行 NPC 行动并交给 Rule Engine 审核 |
| GM 叙事 | `src/engine/aiGM.ts` | 生成开场、回合、事件和结局叙事 |
| 事件触发 | `src/engine/eventTriggerSystem.ts` | 根据 World State 触发 Story Event |
| 章节推进 | `src/engine/storyController.ts` | 根据事件、flag、metric 推进章节 |
| 结局判断 | `src/engine/endingJudge.ts` | 根据 World State 判断结局 |
| 平衡代理 | `src/engine/gmBalancerAgent.ts` | 评估玩家优势，生成软平衡事件 |
| 显示清洗 | `src/engine/displaySanitizer.ts` | 清理内部 id、隐藏指标和剧透信息 |

## 7. 数据流

### 7.1 故事生成链路

```text
用户输入故事创意/完整剧本
  -> normalizeStorySeed
  -> analyzeStoryPlayability
  -> generateStoryBible
  -> applyPlayabilityToBible
  -> validateStoryBible
  -> roomManager.setStoryBible
```

关键 API：

```text
POST /api/stories/generate
POST /api/stories/validate
```

### 7.2 房间与开局链路

```text
创建房间
  -> 加入房间
  -> 选择角色
  -> startStory
  -> createWorldState
  -> 初始化玩家知识、角色状态、NPC Runtime State
```

关键 API：

```text
POST /api/rooms
POST /api/rooms/:roomId/join
POST /api/rooms/:roomId/select-role
POST /api/rooms/:roomId/start
GET  /api/rooms/:roomId/state
```

### 7.3 玩家行动链路

```text
Suggested Action / Free Action
  -> parseSuggestedAction 或 AI parseAction
  -> StructuredAction
  -> canPlayerTakeAction
  -> processPlayerAction
  -> applyUpdates
  -> consumePlayerAction
  -> runDueNPCTurns
  -> runGMBalancerAgent
  -> checkEventTriggers
  -> checkChapterTransition
  -> maybeAdvanceRound
  -> checkEndings + generateGMNarrative
  -> 返回 GM 叙事、推荐行动、指标变化、世界状态
```

关键 API：

```text
POST /api/rooms/:roomId/actions
POST /api/rooms/:roomId/npc-turn
POST /api/rooms/:roomId/endings/check
GET  /api/rooms/:roomId/players/:playerId/view
```

## 9. UI 架构

前端主界面通过 `UIBuilder` 动态渲染 Story Bible 中声明的 widgets。可用 Widget 注册在 `src/registry/widgetRegistry.ts`。

当前支持：

- `NarrativePanel`：GM 叙事
- `ChatPanel`：公共/阵营/私聊频道
- `SuggestedActionsPanel`：推荐行动
- `FreeActionInput`：自由行动输入
- `RolePanel`：当前角色信息
- `MetricPanel`：可见指标
- `EventPanel`：已解锁事件
- `EvidencePanel`：线索与已知事实
- `FactionPanel`：可见阵营信息
- `PlayerInfoPanel`：玩家视角信息

新增 Widget 时需要：

1. 在 `src/components/widgets/` 增加组件。
2. 在 `src/components/widgets/index.ts` 导出。
3. 在 `src/registry/widgetRegistry.ts` 注册 key。
4. 确保 `storyBibleValidator` 能识别该 key。

## 10. AI Provider

AI 接口定义在 `src/types/ai.ts`，实际入口为 `src/lib/aiProvider.ts`。

`src/lib/llmProvider.ts` 提供 OpenAI-compatible Chat Completions 调用，主要能力包括：

- GM 叙事生成
- NPC 行动生成
- 结局叙事生成
- 自由行动解析

当没有 API Key 或调用失败时，部分能力会回退到 `mockAIProvider` 或本地解析逻辑，保证 Demo 可玩。

## 11. 状态存储

MVP 使用 `src/lib/roomManager.ts` 中的内存管理器：

- `rooms: Map<string, Room>`
- `worldStates: Map<string, WorldState>`
- `storyBibles: Map<string, StoryBible>`

这些 Map 挂载在 `globalThis`，可以减少 Next.js 开发模式热更新造成的状态丢失，但不适合生产持久化。后续可替换为 PostgreSQL、Redis 或其他持久层。

4. 选择角色并开始故事。

## 13. 后续开发建议

- 将 Room、WorldState、StoryBible 从内存 Map 迁移到数据库。
- 将 Socket.IO 接入房间状态同步，减少手动刷新。
- 引入更严格的 schema 校验，例如 Zod。
- 为 Rule Engine、Event Trigger、Ending Judge 增加单元测试。
- 为 Story Bible 生成增加版本号和迁移机制。
- 增强观战、掉线重连、房主转移、回放和审计日志。
